## 1. Marktanalyse: KI-Blog-Plattformen 2026

### 1.1 Marktueberblick

Der Markt fuer KI-gestuetzte Content-Erstellung waechst rasant. Die wichtigsten Trends 2026:

1. **Von Content-Generierung zu AI Search Optimization (AEO/GEO)**: Plattformen optimieren nicht mehr nur fuer Google, sondern auch fuer Sichtbarkeit in ChatGPT, Perplexity, Gemini
2. **AI Agents statt Templates**: Enterprise-Plattformen setzen auf autonome KI-Agenten statt statischer Tools
3. **All-in-One vs. Spezialisierung**: Markt spaltet sich in breite Plattformen (GravityWrite) und spezialisierte SEO-Writer (Koala)
4. **Brand Voice als Differenzierer**: Jede ernstzunehmende Plattform bietet Brand-Voice-Training
5. **1-Click Publishing**: WordPress/Shopify/Webflow-Integration ist Standard geworden

### 1.4 Warum sind diese Plattformen erfolgreich?

1. **Sofortiger Wert**: 1 Keyword eingeben → fertiger Artikel in Minuten
2. **SEO-Fokus**: Messbare Ergebnisse (Rankings, Traffic) statt nur "guter Content"
3. **Niedrige Einstiegshuerde**: Free Trials, $9/mo Einstieg, keine Kreditkarte noetig

5. **Bulk-Faehigkeit**: Agenturen brauchen Skalierung (10-100 Artikel gleichzeitig)
6. **Community & Social Proof**: Case Studies, Trustpilot, G2 Reviews, Affiliate Programme

### 2.4 Tech-Schulden & Probleme

**KRITISCH:**
1. **Company Style-Profile NICHT in Blog-Pipeline integriert**: Website wird analysiert, Style wird gespeichert — aber der BlogWriter nutzt das Style-Profile NICHT in den Claude-Prompts. Der USP von Blogreich (Unternehmens-Personalisierung) funktioniert aktuell de facto nicht!
2. **Blog Sections nicht granular gespeichert**: PRD definiert `blog_sections` Tabelle, Code speichert aber alles als Monolith-String in `blogs.content`. Folge: Keine Section-Level-Updates, keine saubere Bild-Zuordnung.
3. **JWKS Cache ohne TTL**: JWT-Keys werden einmalig gecacht, nie refreshed. Bei Key-Rotation durch Supabase bricht Auth.

**HOCH:**
4. **Kein CI/CD**: Keine automatisierten Tests, kein Deployment-Pipeline
5. **Kein Hosting**: Alles nur lokal lauffaehig
6. **PRD nicht aktualisiert**: Alle Features als "geplant" markiert, obwohl einige implementiert sind
7. **Kein Error Monitoring**: Kein Sentry o.ae.
8. **Kein Rate Limiting**: Backend hat keinen API-Rate-Limiter
9. **3 unpushed Commits**: Aenderungen nicht im Remote
10. **FLUX.2 Polling fragil**: `polling_url` vs. Task-ID Handling nicht robust getestet
11. **MagicUI nicht installiert**: In PRD erwaehnt, aber nicht in `package.json`
12. **company_styles / company_style_embeddings**: PRD-Tabellen existieren im Schema-Plan, werden aber nicht genutzt — alles in `company.style_profile` als Text

## 3. Strategie & Positionierung

### 3.1 Blogreich USP (Unique Selling Proposition)

> **"Der einzige KI-Blog-Generator, der wie DEIN Unternehmen schreibt — nicht wie eine KI."**

Blogreich's Kern-Differenzierung ist die **tiefe Unternehmens-Personalisierung**:

1. **Website-Crawling** (Tavily): Analysiert die bestehende Website des Unternehmens
2. **Style-Profil**: LLM extrahiert Tonalitaet, Fachbegriffe, Schreibstil
3. **pgvector Embeddings**: Speichert Style-Daten fuer konsistente RAG-Abrufe
4. **FLUX.2 Section Images**: Hochwertige, kontextbezogene Bilder pro Abschnitt — nicht Stock-Fotos

**Warum kein Wettbewerber das bietet:**
- GravityWrite/Koala: Nur "Tone of Voice" Dropdown (formell/locker)
- Jasper: Brand Voice Training, aber manuell (User muss Texte hochladen)
- Blogreich: **Automatische** Website-Analyse → sofortige Personalisierung

### 3.2 Zielgruppen-Fokus (3 Personas)

#### Persona 1: "Marketing-Mia" — KMU Marketing-Managerin (DACH)
| Attribut | Details |
|----------|---------|
| **Rolle** | Marketing-Managerin in einem DACH-KMU (10-100 MA) |
| **Alter** | 28-42 |

### 3.3 Positionierungsstrategie

**Nicht breit targeten, sondern spitz starten:**

**Phase 1 (MVP):** Fokus auf **Persona 1 (Marketing-Mia)** im DACH-Raum
- Deutscher Content ist weniger abgedeckt als englischer
- DACH-KMU haben Budget aber wenige gute KI-Tools auf Deutsch
- Website-Personalisierung ist fuer KMU-Marketing am wertvollsten

**Phase 2 (Scale):** Erweiterung auf **Persona 2 (Agentur-Alex)**
- Multi-Company Support ist bereits architektonisch vorhanden
- Agenturen bringen hoehere ARR und laengere Retention

**Phase 3 (Expand):** **Persona 3 (Solo-Stefan)** + internationaler Markt
- Easy Blog Writer Modus fuer Solopreneure
- Englischer Markt mit USP "Company-Personalized AI Blogs"

#### 4.1.4 AWS Hosting Setup
**Fuer:** Mensch (Adrian) + Claude Code (Dockerfile, CI/CD)

**WICHTIG: AWS App Runner nimmt ab 30. April 2026 keine neuen Kunden mehr an!**
AWS empfiehlt offiziell die Migration zu **Amazon ECS Express Mode** (seit Nov 2025).

**Empfohlene Architektur: AWS Amplify (Frontend) + ECS Express Mode (Backend)**

```
                    Route 53 (DNS)
                         |
              ┌──────────┴──────────┐
              |                     |
        AWS Amplify          ECS Express Mode
      (React SPA + CDN)      (FastAPI Backend)
      Git-Push = Deploy       Docker Container
      Auto SSL, Preview       Auto-Scaling, ALB
              |                     |
              └──────────┬──────────┘
                         |
                   Supabase (extern)
                   (Auth, DB, Storage)
```

**Optionsvergleich:**

| Aspekt | D': Amplify + ECS Express | B: ECS Classic | C: EC2 |
|--------|:--:|:--:|:--:|
| **Setup-Zeit** | 30-60 Min | 4-8 Std | 2-4 Std |
| **Kosten (Start)** | ~15-35 EUR/mo | ~25-60 EUR/mo | ~5-15 EUR/mo |
| **Kosten (1000+ User)** | ~40-130 EUR/mo | ~60-200 EUR/mo | Umbau noetig |
| **Komplexitaet** | Niedrig | Hoch | Mittel (Ops) |
| **Auto-Scaling** | Ja | Ja | Nein |
| **CI/CD** | Frontend: eingebaut | Separat | Manuell |
| **HTTPS/SSL** | Automatisch | Manuell (ACM) | Manuell (Certbot) |
| **Zukunftssicherheit** | Hoch | Sehr hoch | Niedrig |

**Warum Amplify + ECS Express Mode:**
- **ECS Express Mode** ist der offizielle App-Runner-Nachfolger, gleiche Einfachheit
- Ein CLI-Befehl fuer Backend-Deployment, automatisch ALB, HTTPS, Auto-Scaling
- Bei Wachstum: Nahtlose Migration zu klassischem ECS Fargate (gleiche Infrastruktur)
- **Amplify**: Git-Push = Deployment, Preview-Branches, SSL, CDN — zero Config
- Externe DB (Supabase) entlastet AWS-Architektur — Backend ist reiner API-Server

**Geschaetzte AWS-Kosten (Monatlich, 10-100 User):**
| Service | Kosten |
|---------|--------|
| ECS Express Mode (0.25 vCPU, 0.5GB RAM) | 10-25 EUR/mo |
| Shared ALB (bis zu 25 Services) | 5-10 EUR/mo |
| AWS Amplify Hosting (Frontend) | 0.50-3 EUR/mo |
| ECR (Container Registry) | ~1 EUR/mo |
| Route 53 (DNS) | ~0.50 EUR/mo |
| **Gesamt** | **~15-35 EUR/mo** |

**Deployment-Steps:**
1. Dockerfile fuer Backend erstellen (FastAPI + uvicorn)
2. ECR Repository anlegen, Docker Image pushen
3. ECS Express Mode Service konfigurieren (ENV Vars, Port 8123)
4. Amplify App verbinden (GitHub Repo → `project/` Ordner)
5. Route 53 Custom Domain konfigurieren
6. GitHub Actions CI/CD Pipeline fuer Backend

**Relevante AWS-Dokumentation:**
- ECS Express Mode: docs.aws.amazon.com/AmazonECS/latest/developerguide/express-service-overview.html
- Amplify Hosting: aws.amazon.com/amplify/hosting/
- App Runner Deprecation: docs.aws.amazon.com/apprunner/latest/dg/apprunner-availability-change.html
- Migration Guide: aws.amazon.com/blogs/containers/migrating-from-aws-app-runner-to-amazon-ecs-express-mode/
